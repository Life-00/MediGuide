# 🤝 Contributing Guide

**MediGuide**에 기여해주셔서 감사합니다!  
이 문서는 효율적인 협업을 위한 **작업 흐름, 협업 규칙, 커밋 컨벤션**을 정리한 가이드입니다.

> ❗ **본 프로젝트는 모든 작업을 Pull Request(PR) 기반으로 진행합니다.**

---

## 🌿 브랜치 전략

### 🔒 **main 브랜치 규칙**
* `main` 브랜치 **직접 push 절대 금지**
* 모든 변경 사항은 **PR을 통해서만 병합** 가능
* 최소 **1명 이상의 리뷰 승인** 필수

### 🌱 **브랜치 네이밍 규칙**
형식: `type/작업-내용`

| 타입 | 설명 | 예시 |
| :--- | :--- | :--- |
| **feature** | 새로운 기능 개발 | `feature/login-api` |
| **fix** | 버그 수정 | `fix/token-expire` |
| **docs** | 문서 작업 (README 등) | `docs/readme-update` |
| **refactor** | 코드 리팩토링 | `refactor/cleanup` |
| **chore** | 설정, 빌드, 라이브러리 관리 | `chore/github-actions` |

---

## 🔄 작업 흐름 (Workflow)

1. **main 브랜치 최신화**: `git pull origin main`
2. **작업 브랜치 생성**: `git checkout -b type/작업-명칭`
3. **기능 개발**: 로컬에서 작업 진행
4. **커밋**: 컨벤션에 맞춰 커밋 메시지 작성
5. **Pull Request 생성**: 작업 완료 후 GitHub에서 PR 생성
6. **코드 리뷰**: 동료의 리뷰 및 피드백 반영
7. **승인 후 병합**: 승인 완료 시 `main` 브랜치로 병합

---

## 📝 커밋 컨벤션

### 📌 **커밋 메시지 형식**
`type: 내용` (영문/국문 혼용 가능하나 명확하게 작성)

* ✨ **feat**: 새로운 기능 추가
* 🐛 **fix**: 버그 수정
* 📝 **docs**: 문서 수정
* 🎨 **style**: 코드 포맷팅 (코드 로직 수정 없음)
* ♻️ **refactor**: 리팩토링
* 🔧 **chore**: 빌드 설정, 패키지 관리

> **💡 예시**: `feat: 사용자 로그인 API 구현`, `fix: 토큰 만료 처리 로직 수정`

---

## 🔍 Pull Request 및 리뷰 규칙

### ✅ **PR 체크리스트**
* [ ] **PR 템플릿** 내용 작성 완료
* [ ] 로컬에서 **빌드 및 실행 테스트** 완료
* [ ] 불필요한 설정 파일/로그 포함 여부 확인
* [ ] **커밋 컨벤션** 준수 여부

### 👀 **리뷰 규칙**
* 최소 **1명 이상**의 승인이 있어야 병합 가능합니다.
* 리뷰 반영 후 다시 리뷰를 요청하세요.
* **본인 PR을 본인이 승인**하는 것은 금지합니다.

---

## 🚫 금지 사항
* **main 브랜치 직접 push** ❌
* 히스토리를 파괴하는 **force push** ❌
* 테스트되지 않은 코드의 **PR 생성** ❌
* 정해진 **협업 규칙 및 컨벤션 무시** ❌

---

**강강술래 팀의 깔끔한 협업과 빠른 개발을 위해 함께 지켜주세요!** 🚀
